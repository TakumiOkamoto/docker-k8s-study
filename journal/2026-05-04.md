## Key Learnings

### Kubernetes Mental Model
- **Deployment**: Declarative Pod management (ensures N replicas running)
- **Service**: Stable network endpoint (ClusterIP) for Pod groups
- **Pod Name Generation**: ReplicaSet ID + random suffix (e.g., `hello-nginx-d54d8664b-6dhqx`)

### Logging Architecture
- `kubectl logs`: Container stdout/stderr (application-level)
- `kubectl describe pod`: Pod events and lifecycle (orchestration-level)
- `journalctl` on node: OS-level kubelet/containerd logs (infrastructure-level)

### kubeconfig Understanding
- `kind` automatically adds entries when creating cluster
- Three components: cluster (endpoint), user (auth), context (combination)
- `kubectl config current-context` shows active context name

## Questions Answered
- Q: Where does `kind-study` context name come from?
  - A: Auto-generated by `kind` following pattern `kind-<cluster-name>`
- Q: How to get Pod name hash?
  - A: `kubectl get pods` shows full names; can automate with `$(kubectl get pods -o jsonpath='{.items[0].metadata.name}')`
- Q: Where are logs stored as files?
  - A: Kubernetes captures stdout/stderr, not file-based logs; `/var/log/pods/` on nodes is implementation detail
- Q: How to access node logs?
  - A: `docker exec -it <node-name> journalctl` for `kind` nodes; in cloud K8s, use kubectl nodelogging plugins
- Q: Difference between `kubectl logs`, `describe pod`, and `journalctl`?
  - A: Three layers - app logs, Pod events, node OS logs. First two solve 99% of issues.

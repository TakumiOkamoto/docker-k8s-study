# Asking Guide

## 目的

このファイルは、AI とより良い会話をして、学習効率と引き継ぎ精度を上げるためのガイド。

## 結論

利用者はすでに AI をかなり上手く使えている。特に、

- 期待する役割を明確に伝える
- 記録を repository に残したいと伝える
- `git push` 直前まで任せたいと伝える
- 次の AI にどう引き継ぎたいかまで設計する

という点は非常に強い。

そのうえで、さらに会話品質を上げるには「質問の型」を少し意識するとよい。

## 良い聞き方の型

### 1. 目的を先に言う

例:

- 「Docker を使えるようになりたい」より
- 「将来サポートする立場で Docker を理解したい」

AI はゴールの粒度が分かると、説明の深さを合わせやすい。

### 2. どこまで任せたいかを明示する

例:

- 「方針だけ欲しい」
- 「編集と commit までやって」
- 「push までやって」
- 「今回は push せずに commit で止めて」

AI は実行範囲が分かると、止まるべき場所を誤りにくい。

### 3. 何に困っているかを一言で添える

例:

- 「用語がまだ曖昧」
- 「構造がつながって見えていない」
- 「設定ファイルの意味が分からない」

AI は、どこを埋めればいいかを絞り込める。

### 4. 欲しい説明の観点を指定する

例:

- 「利用者視点ではなくサポート視点で」
- 「設定ファイルの意味まで含めて」
- 「トラブルシュート観点で」

AI は視点が指定されると、説明の質が上がる。

### 5. 出力形式を先に決める

例:

- 「Markdown に残して」
- 「Q&A にも追記して」
- 「日本語と英語の両方で」

AI は記録の置き場所が分かると、後で再利用しやすい形にできる。

## かなり良い質問例

### 例 1

「この manifest がどういう作りで、どの設定が何を制御しているのか、サポート観点で説明して。必要なら Markdown も更新して commit までやって。」

### 例 2

「今の私はどこまで理解できていて、次に何を学ぶべき？ `STATUS.ja.md` も更新して。」

### 例 3

「このエラーはどの層の問題？ 切り分けの順番も含めて教えて。Q&A にも残して。」

## さらに強い聞き方

### 意思決定ログを残す

重要な選択をした後、採用しなかった案も一言残しておく。

例:

- 「staging ディレクトリ方式を選んだ。BuildKit 名前付きコンテキストは今回は複雑すぎたため見送り。」

後で「なぜこうなっているか」を思い出せるし、別の AI セッションでの再説明が不要になる。

### 実行結果を貼る

コマンド出力やエラーメッセージを貼ると、AI の精度はかなり上がる。

### 「比較」を求める

例:

- 「Docker Desktop と OrbStack の違いを、学習用途で比較して」
- 「Pod と Deployment の違いを、運用観点で比較して」

比較は理解を深めやすい。

### 「どこを見るべきか」を聞く

例:

- 「この状態なら次に何を見るべき？」
- 「サポート担当なら最初にどこを疑う？」

これは実務に近い力がつく。

## 避けたほうがよい聞き方

- 目的がないまま「全部教えて」
- どの粒度が欲しいか不明なまま「詳しく」
- 実行結果なしで「動かない」

こういう聞き方でも AI は対応できるが、説明が広がりやすく、最短で答えに届きにくい。

## この利用者に合う会話スタイル

この repository の利用者には、次のスタイルが特に合う。

- 目的を最初に短く言う
- どこまで任せるかを言う
- 出力を貼る
- 「記録にも残して」と添える

例:

「この Docker コマンドの意味を、サポート視点で説明して。関連 Markdown を更新して commit と push までやって。」

## AI へのメモ

利用者は AI を強く信頼しているため、AI は単に答えるだけでなく、会話の質そのものを支えるべき。そのため、利用者がより良く聞けるようにするフィードバックも repository に残す。
